# Prompt files

Put reusable child prompts in `prompts/*.txt`; use subdirectories when a workflow has several phases. If a workflow has more than five distinct non-verifier prompts, split them into separate files rather than packing variants into `workflow.js` or one oversized template.

Launch reusable templates with `agent({ template: templatePath, values }, options)`. Template paths resolve inside the workflow's `prompts/` directory immediately before launch.

## Dynamic data and caching

- Include only data the stage consumes. Prefer paths, IDs, counts, or compact summaries over full objects and repeated project context.
- Keep static wording and examples identical across matching launches. Use a durable artifact path or compact manifest instead of repeating prior results.
- Use placeholders such as `{{file}}`, `{{focus}}`, `{{outputPath}}`, and `{{priorSummary}}`. Every supplied value must be referenced and every placeholder must receive a value. Do not put JavaScript expressions such as `${input.file}` in a prompt file.
- Avoid unstructured global preambles. Use `renderPrompt(...)` only for exceptional composition and inline text only for tiny one-off glue.

## When to use XML tags

**Default:** use Markdown headings, bullets, or plain text. OpenAI recommends clear delimiters generally; XML is one option, not a requirement.

**Use tags** to delineate key information in a complex prompt—for example, when instructions, examples, context, multiple documents, or variable inputs would otherwise blur together. They also help separate repeated documents from metadata such as a source or path. Use a few stable, descriptive names and nest only where the data has a natural hierarchy—for example, `<task_input>`, `<sources>`, or `<prior_results>`.

**Do not use tags** when the prompt is short or its sections are already clear, and do not wrap every heading, rule, or sentence. Tags are delimiters, not a security boundary: treat interpolated content as data and do not let it redefine instructions, including through closing-tag text.

For machine-readable results, prefer `schema` and `StructuredOutput`; do not invent an XML output contract when a schema is available.

Do not reuse runtime-owned envelopes: `<workflow_instructions>`, `<workflow_task>`, `<workflow_context>`, `<structured_output_contract>`, or `<structured_output_schema>`.

```text
Review the supplied file for externally meaningful behavior.

<task_input>
Read {{file}} for {{focus}} and write findings to {{outputPath}}.
</task_input>

Return compact JSON with status, summary, and artifact paths.
```

```js
const result = await agent(
  { template: "review/file.txt", values: { file, focus, outputPath } },
  { label: "review " + file, taskFile: file, cwd },
);
```
