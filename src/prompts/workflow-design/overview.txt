# Workflow overview

Use an existing workflow when it fits. Author a new one only when the request is reusable or multi-step enough to need a runbook.

- Before asking about missing details, inspect project context and infer the purpose, inputs/defaults, phases, child-agent roles, file reads, and result shape when reasonably clear.
- Draft under `{{draftRoot}}`, then call `propose_workflow` with the workflow name.
- Keep `workflow.js` for orchestration. Put reusable child prompts in `prompts/*.txt` and launch them with `agent({ template, values }, options)`.
- Use `LLM` for one generation-only response without tools or a child session. Use `agent` when the model needs tools, iterative work, or persistent child-agent state.
- Give agent-calling primitives explicit `extensions` or `tools` lists only when they should not inherit project defaults. `[]` means none; a named extension tool loads its owner without exposing sibling tools.
- Keep static prompt instructions and examples stable. Pass only stage-specific paths, IDs, counts, or compact manifests as dynamic data.
- Ask for `workflow-api`, `draft-directory`, `prompt-files`, `child-agents`, `structured-outputs`, `fanout`, `verification`, or `artifacts` only when that focused guidance is needed.
