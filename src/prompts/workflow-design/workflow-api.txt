<workflow_guidance topic="workflow-api">
<purpose>Use the sandbox globals and metadata contract in workflow.js.</purpose>
<rules>
<rule>workflow.js exports static metadata and a default async workflow function; metadata must include name, description, inputInstructions, and phases.</rule>
<rule>Document the default function with JSDoc covering purpose, input fields/defaults, phases, child agents, file reads, and result shape.</rule>
<rule>Supported workflow primitives are rendered below from the runtime registry with short descriptions and signatures; call narrower guidance topics for examples only when needed.</rule>
<rule>agent, mapreduce, and verifier accept extensions?: string[] and tools?: string[]. Omit to inherit workflow settings, use [] for none, and use an exact tool list when only named tools should be exposed. A schema-enabled agent always receives its terminal StructuredOutput tool.</rule>
<rule>propose_workflow statically validates inline and top-level const capability lists plus inherited defaults against Pi extension/tool metadata and saves nothing on error.</rule>
<rule>Receive workflow input through workflow({ field, optional = default }) parameters.</rule>
<rule>Workflows cannot import modules or use ambient Node globals. Use readText/readJson/writeText/writeJson and workflow-owned template tasks for files and artifacts.</rule>
</rules>

  name: "review",
  description: "Review selected files",
  inputInstructions: "Resolve files from explicit path mentions; use remaining prose as focus.",
  phases: [{ title: "review" }, { title: "synthesis" }],
};
</example>
