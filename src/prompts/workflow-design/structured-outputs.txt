<workflow_guidance topic="structured-outputs">
<purpose>Require semantic structured results without parsing assistant text.</purpose>
<rules>
<rule>Use agent({ template, values }, { schema }) when a child must return structured fields.</rule>
<rule>The schema is prepended to the child prompt in a &lt;structured_output_schema&gt; section and becomes the terminal StructuredOutput tool parameter contract. Describe each field's meaning inline with JSON Schema descriptions.</rule>
<rule>StructuredOutput validates its keyword arguments, captures them, and ends the child session. A schema-enabled child that does not call it fails.</rule>
<rule>The agent result always includes message, name, steps, and usage. message is null when a terminal call omits it; name, steps, and usage are runtime-owned and cannot be schema properties.</rule>
<rule>Keep schemas shallow and semantic: required fields, useful descriptions, enums/booleans, and additionalProperties: false. Add bounds only for real downstream limits.</rule>
<rule>Use JSON as the control surface only: status, decision, IDs, counts, short summary, artifactPaths, evidence paths, and line references.</rule>
<rule>Route large reasoning, transcripts, diffs, reports, and generated artifacts to named files; return their paths and a compact manifest.</rule>
<rule>For large lists, prefer capped summaries or JSONL/artifact files over one giant nested JSON object.</rule>
<rule>Stage expansion: first return a compact selector/manifest, then launch follow-up agents only for selected items that need detail.</rule>
<rule>Use stable IDs and lookup tables instead of repeating long strings across every finding.</rule>
</rules>
<example>
const schema = {
  type: "object",
  properties: { status: { type: "string", enum: ["pass", "fail"] }, summary: { type: "string" } },
  required: ["status", "summary"],
  additionalProperties: false,
};
</example>
</workflow_guidance>
